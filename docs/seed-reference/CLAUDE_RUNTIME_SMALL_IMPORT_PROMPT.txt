THE OTHER 99 — USE SMALL V3 RUNTIME SEED PACKAGE

Use this package:
the_other_99_SEMANTIC_SEED_DATABASE_v3_RUNTIME_SMALL.zip

This is the small implementation package.

Do NOT use the oversized full v3 ZIP in the frontend bundle.
Do NOT import the 600,000 raw synthetic votes into the frontend runtime.
The app only needs projected percentages and real vote persistence.

==================================================
WHY THIS SMALL PACKAGE EXISTS
==================================================

The full v3 package is too large for normal repo/frontend work.

This small package contains:
- projected percentages,
- compact JSON runtime data,
- scenario comparison data,
- content vectors,
- mapping file,
- archetype definitions,
- calibration targets,
- SQL template.

It intentionally excludes:
- raw 600k votes,
- full synthetic population,
- full synthetic user profiles.

==================================================
DEFAULT BASELINE
==================================================

Use:

projected_percentages_early_app_userbase.json

as the default runtime baseline.

Default scenario:

early_app_userbase_projected

Reason:
The Other 99 is a self-discovery app. Its early users are more likely to resemble an introspective app audience than the full general population.

==================================================
CRITICAL MAPPING RULE
==================================================

Do NOT use semantic IDs as production content IDs.

Bad:
semantic_v3_q0001

Correct:
Map semantic_v3_q0001 to an existing real app content_id using:

content_id_mapping_to_real_app_REQUIRED.csv

Only mapped rows can be used in production.

If mapping cannot be automated:
- create/update the mapping file,
- list unmapped items,
- ignore unmapped rows at runtime,
- mark mapping as PARTIAL in docs/IMPLEMENTATION_CHECKLIST.md.

==================================================
RUNTIME FORMAT
==================================================

Use:

projected_percentages_early_app_userbase.json

This contains entries like:

{
  "semantic_v3_q0001": {
    "source": "seed_semantic_v3",
    "scenario_id": "early_app_userbase_projected",
    "distribution_label": "Projected distribution",
    "total_seed_votes": 400,
    "answers": {
      "A": { "count": 120, "pct": 0.3000 },
      "B": { "count": 170, "pct": 0.4250 },

      "SKIP": { "count": 5, "pct": 0.0125 }
    },
    "behavioral_averages": {

Before using in the app, transform keys from semantic IDs to real app content_id.

==================================================
IMPLEMENTATION REQUIREMENTS
==================================================

1. Add the small seed package to the repo.
2. Do not include raw synthetic votes in frontend runtime.
3. Build or update a mapper:
   semantic_content_id -> real content_id.
4. Create lightweight runtime projected distribution data keyed by real content_id.
5. Use early_app_userbase_projected as default baseline.
6. Keep other scenarios available for DebugPanel preview if feasible.
7. Every real answer submitted in the app must save/update a real vote.
8. Real votes must gradually replace projected seed data.
9. DebugPanel must show:
   - seed/projected vote count,
   - real vote count,
   - scenario_id,
   - distribution label,
   - whether current content_id is mapped.

==================================================
LABEL RULES
==================================================

If only projected seed data exists:
Projected distribution

If real_votes >= 30 and < 100:
Early community distribution

If real_votes >= 100:
Community distribution

Never label seed-only data as:
- How others answered
- Real users answered
- Community answered
